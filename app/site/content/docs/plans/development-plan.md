> 本文属于“任务计划”文档，描述当前仍在推进中的阶段目标与执行优先级。

当前项目的**核心流程**与**数据架构**已经基本稳定。虽然仍然存在一些细节问题和局部不一致，但整体方向已经明确，具备进入下一阶段系统化优化与功能深化的条件。

下一阶段将围绕“**先夯实结构，再提升体验，最后补足能力**”的思路推进，整体优先级如下：

1. **项目结构优化**
2. **交互性优化**
3. **提示词优化**
4. **剪辑功能完善**

## 阶段目标

接下来的开发重点，不再是继续快速堆叠功能，而是转向以下几个方向：

- 提升代码可读性与可维护性
- 收敛数据模型与接口规范
- 优化关键流程中的交互体验
- 提高多 Agent 场景下提示词的稳定性与产出质量
- 在不过度设计的前提下，逐步补齐剪辑能力

换句话说，项目后续会更加重视“**稳定、清晰、可持续演进**”，而不仅仅是“功能可用”。

## 开发优先级总览

### 1. 项目结构优化

这是当前阶段的**最高优先级**。

由于项目主要由个人开发完成，其中包含较多 AI 辅助生成代码。虽然这种方式提升了早期开发速度，但也带来了一些典型问题：

- 代码复用性不足
- 命名和职责边界不够统一
- 局部实现可读性较差
- 部分层级存在职责混杂
- 数据关联关系偏多，后续维护成本较高

因此，在继续扩展功能之前，必须优先把项目结构整理清楚。

#### 优化方向

**数据模型标准化**

- 梳理现有数据模型
- 减少不必要的关联关系
- 收敛重复字段和重复语义
- 让核心业务对象边界更清晰

**Service 层标准化**

- 明确 service 的职责边界
- 避免把业务逻辑散落在 API 层或模型层
- 提升 service 层复用性
- 让复杂流程有稳定的承载位置

**API 层优化**

- 标准化接口入参
- 标准化接口返回结构
- 统一错误响应和状态表达
- 减少接口行为不一致的问题

**目录结构梳理**

- 明确各目录职责分工
- 减少“看名字不清楚做什么”的情况
- 提升代码定位与协作效率

**无用代码清理**

- 清理遗留的试验性实现
- 删除重复逻辑和低价值兼容代码
- 降低维护负担

### 2. 交互性优化

交互优化是第二优先级。

当前项目在主流程上已经“基本可用”，但某些关键步骤的交互仍然不够直观，尤其是：

#### 剧本资产信息提取流程

从剧本中提取角色、场景、服装、道具等资产信息这一环节，目前虽然可以使用，但整体体验仍然比较勉强，主要问题包括：

- 信息提取过程不够清晰
- 用户难以快速判断提取结果是否可靠
- 编辑、确认、修正的链路还不够顺畅
- 结果与后续分镜、资产管理之间的衔接还可以更自然

#### 后续优化重点

- 提高资产提取流程的可视化程度
- 优化提取结果的展示方式
- 增强人工确认与修正的便捷性
- 强化提取结果与后续页面之间的衔接体验

这一部分的目标不是单纯“更好看”，而是让关键业务步骤更**可理解、可确认、可操作**。

### 3. 提示词优化

提示词优化是第三优先级，但对项目产出质量影响很大。

当前项目已经包含多个 Agent，不同 Agent 分别承担剧本处理、资产提取、图片生成、关键帧生成等任务。下一阶段需要系统性优化这些提示词，以提升：

- 输出稳定性
- 结果一致性
- 可控性
- 与业务流程的贴合程度

#### 重点优化范围

**图片生成相关提示词**

- 演员图片生成提示词
- 场景图片生成提示词
- 服装图片生成提示词
- 道具图片生成提示词
- 角色图片生成提示词

**关键帧相关提示词**

- 首尾关键帧图片文案提示词
- 首尾关键帧图片生成提示词

**剧本处理相关提示词**

- 剧本优化提示词
- 剧本资产信息提取提示词

#### 优化目标

- 让提示词更贴近项目实际业务场景
- 提高同一项目内生成结果的一致性
- 减少无效输出和偏题输出
- 让不同 Agent 之间的上下文衔接更稳定

这一阶段的重点不是让提示词变得更复杂，而是让提示词真正服务于产品流程。

### 4. 剪辑功能

剪辑功能的优先级最低。

原因比较明确：

- 剪辑本身偏前端能力
- 复杂剪辑能力很难超过专业编辑软件
- 当前项目已经具备导出能力
- 从整体产品价值来看，剪辑不是现阶段最核心的差异点

因此，后续对剪辑功能的投入会保持克制，更倾向于满足“**够用、顺畅、能衔接导出**”的目标，而不是朝专业视频编辑器方向演进。

#### 当前判断

剪辑功能会继续保留并逐步完善，但不会在近期成为主要开发重点。

#### 后续方向

- 优化已有剪辑流程的基础体验
- 保证与素材、时间线、导出链路的稳定衔接
- 控制功能边界，避免过度投入到专业编辑能力竞争中

## 整体开发策略

后续开发将遵循以下原则：

### 先治理，再扩展

在结构未整理清楚之前，不盲目增加复杂功能。

### 先标准化，再做体验优化

接口、数据模型、service 分层先收敛，交互优化才更容易稳定落地。

### 提示词优化服务于业务，而不是独立存在

提示词不会单独作为“研究项”推进，而是围绕具体业务节点逐步优化。

### 剪辑功能保持边界意识

坚持“实用优先”，避免投入过多精力去复制专业软件能力。

## 阶段性里程碑建议

### 第一阶段：结构治理

- 数据模型标准化
- 接口返回值标准化
- service 层规范化
- 目录结构整理
- 清理无用代码

### 第二阶段：流程体验优化

- 优化资产提取流程交互
- 提升关键页面的信息组织和操作体验
- 收敛主流程中的易混乱节点

### 第三阶段：提示词专项优化

- 梳理多 Agent 提示词职责
- 分模块优化图片、关键帧、剧本相关提示词
- 提升结果稳定性与一致性

### 第四阶段：剪辑能力补强

- 补充基础可用能力
- 优化导出前的使用体验
- 保持轻量化和边界清晰

## 总结

项目已经从“快速搭建原型”进入“系统化打磨”的阶段。

接下来的重点，不是继续无序扩张，而是围绕以下主线推进：

- **把结构理顺**
- **把流程做顺**
- **把提示词做稳**
- **把剪辑做成够用的辅助能力**

这会让项目从“可以运行”逐步走向“更稳定、更清晰、更适合长期演进”。
